# ⚡ FaultSense — Turbofan Engine Predictive Maintenance

> Remaining Useful Life (RUL) estimation for jet engines using LSTM on the NASA C-MAPSS dataset, with a real-time Plotly Dash fleet health dashboard.

---

## Overview

FaultSense is an end-to-end predictive maintenance system for turbofan engines. It trains an LSTM model on NASA's C-MAPSS benchmark dataset to predict how many flight cycles remain before an engine requires maintenance, then visualises fleet health in a production-ready interactive dashboard.

The system classifies each engine into one of three operational tiers — mirroring real maintenance protocols:

| Tier | RUL | Action |
|------|-----|--------|
| 🔴 **CRITICAL** | < 30 cycles | Ground immediately · Emergency maintenance |
| 🟡 **WARNING** | 30–80 cycles | Plan maintenance within next mission cycle |
| 🟢 **HEALTHY** | > 80 cycles | Continue normal operations |

---

## Results

Performance on NASA C-MAPSS FD001 test set (100 engines):

| Metric | Value |
|--------|-------|
| Test RMSE | ~16–20 cycles |
| NASA Score | lower = better (asymmetric penalty) |
| Model parameters | ~1,400 |
| Training time (CPU) | < 5 min |

The NASA scoring function penalises late predictions (predicting more RUL than remains) more steeply than early ones, reflecting that missing a fault is more dangerous than a false alarm:

```
d = pred − true
S = Σ exp(-d/13) - 1   if d < 0  (late, steep)
    Σ exp( d/10) - 1   if d ≥ 0  (early, mild)
```

## Quickstart

### 1 — Preprocess the data

```bash
python preprocess.py --subset FD001
```

This runs the full pipeline:
- Drops near-zero-variance sensors (s1, s5, s10, s16, s18, s19 for all subsets; additionally s6 for FD001)
- Applies piecewise-linear RUL labelling capped at 130 cycles
- Fits z-score normalisation on training data only (no leakage)
- Splits 80/20 by engine ID
- Builds sliding windows of 30 cycles with stride 1
- Zero-pads short test engines
- Saves engine IDs per window for trajectory visualisation

Output: `data/processed/cmapss_fd001.pkl`

### 2 — Train the model

```bash
python train.py --subset FD001
```

Training spec (matching paper):
- Optimiser: RMSprop, lr=0.001, weight_decay=1e-5
- Loss: MSE
- Batch size: 64
- Scheduler: ReduceLROnPlateau (factor=0.5, patience=5)
- Early stopping: patience=20 on val loss
- Max epochs: 200
- Seed: 42

MLflow tracking runs at `http://localhost:5000` by default (falls back to `./mlruns`).

To train all 4 subsets:

```bash
python train.py --all
```

### 3 — Launch the dashboard

```bash
python app.py
```

Open `http://127.0.0.1:8888` in your browser.

## Model Architecture

LSTM for RUL estimation (paper Algorithm 5):

```
Input: (B, 30, n_sensors)
  └─ LSTM(hidden=32, batch_first=True)
       └─ take last hidden state hT  →  (B, 32)
            └─ Dropout(0.5)
                 └─ Linear(32→8) → ReLU
                      └─ Linear(8→8) → ReLU
                           └─ Linear(8→1)
                                └─ clamp(0, 1)   [× rul_cap at inference]
```

Targets are normalised to [0, 1] during training (`y / rul_cap`). The model output is multiplied by `rul_cap` at inference time to get predictions in cycles.

Total trainable parameters: **~1,400**

## Preprocessing Details

### Sensor selection

| Dropped (all subsets) | Dropped (FD001 only) | Reason |
|-----------------------|----------------------|--------|
| s1, s5, s10, s16, s18, s19 | s6 | Near-zero variance — no degradation signal |

FD003 retains s6 (correlation −0.215 with RUL; nonlinear spread near failure under dual fault mode).

### Normalisation

Z-score normalisation fitted on training fleet only:

```
x_norm = (x - μ_train) / σ_train
```

For FD002/FD004 (6 operating conditions), statistics are fit per condition cluster (K-means, k=6) to remove regime offsets while preserving degradation signal.

### RUL labelling

Piecewise-linear (PwL) with cap = 130:

```
RUL(t) = min(max_cycle - t, 130)
```

Early healthy cycles are assigned the constant value 130, preventing the model from learning spurious degradation signals during stable operation.

## Dashboard Features

### Fleet Overview
- Horizontal bar chart of all engine RULs, colour-coded by tier
- Click any bar to select that engine
- Three tier badges showing fleet-wide counts

### Engine Inspector (sidebar)
- Number input — type any engine index directly
- Prev / Next buttons for stepping through engines
- Jump buttons — skip to next Critical / Warning / Healthy engine
- Per-engine detail card: status, predicted RUL, true RUL, error, maintenance action

### RUL Trajectory
- Full lifecycle plot for the selected engine (sliding-window inference on training data)
- Predicted vs true RUL over all cycles
- Coloured zone bands (CRITICAL / WARNING / HEALTHY)
- Star marker showing the test set prediction point

### Predicted vs True Scatter
- All 100 test engines plotted against the perfect-prediction diagonal
- Colour = |error|, star = selected engine

### Error Distribution
- Histogram of prediction errors with late/early shading

### NASA Penalty Chart
- Per-engine asymmetric penalty, sorted descending

### Cost Analysis
- Live cost modelling with adjustable parameters (sidebar)
- Three scenarios per engine: unplanned failure risk, unnecessary maintenance, optimal scheduling
- Total fleet cost, predicted savings vs reactive baseline
- Failure-risk engine table with immediate action flags

## Baseline Comparison — UKF

`ukf.py` implements an Unscented Kalman Filter baseline for anomaly detection. The dashboard's LSTM vs UKF tab compares:

- Precision, Recall, F1
- Mean detection lead time (cycles before fault)
- Early detection rate (≥ N cycles lead time)
- False alarm rate (per 100 healthy cycles)

The LSTM advantage: 30-cycle context window captures gradual correlated drift across multiple sensors that the UKF adapts to and misses. UKF advantage: reacts to sudden step-change faults in 1 cycle vs 3–5 for LSTM.

## Configuration

Key hyperparameters in `train.py`:

```python
DEFAULT_CONFIG = {
    "hidden":        32,      # LSTM hidden units
    "dropout":       0.5,     # Dropout on last hidden state
    "rul_cap":      130.0,    # RUL cap (must match preprocess.py)
    "epochs":       200,      # Max training epochs
    "lr":           1e-3,     # RMSprop learning rate
    "weight_decay": 1e-5,     # RMSprop weight decay
    "batch_size":    64,
    "patience":      20,      # Early stopping patience
    "sched_factor":  0.5,     # LR reduction factor
    "sched_patience": 5,      # LR scheduler patience
    "oversample":    2.0,     # Weight multiplier for high-RUL windows
}
```

Per-subset overrides can be added to `SUBSET_CONFIG`.
